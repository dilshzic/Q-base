EXEC1. Executive Summary & Vision

Q-base is an unprecedented, natively built Android platform targeting the complex domain of medical education. Unlike traditional flashcard applications (e.g., Anki, Quizlet), Q-base introduces a fundamentally multi-modal, AI-first paradigm engineered for absolute data ownership, peer-to-peer encrypted collaboration, and offline survivability.

The application spans tens of thousands of lines of robust Kotlin code, intertwining massive Room SQLite databases with real-time Firebase listeners, Appwrite bucket blob storage, and advanced Gemini LLM integrations. The Brain Engine serves as the central nervous system, intelligently routing complex unstructured medical texts into highly structured JSON formats, subsequently hydrating the 14-table localized relational database.

To ensure uncompromising privacy in an era of data brokering, Q-base integrates Google Tink to guarantee mathematically verifiable End-to-End Encryption (E2EE) for all social communication channels and file transfers. The cloud provider mathematically cannot read the plaintext of the medical collections shared among colleagues.

This document acts as the definitive source of truth, covering the application across its 5 major planes: The Compose UI Layer, the Offline SQLite Data Layer, the AI Abstractive Layer, the Cloud Transport Layer, and finally the Defensive Cryptographic Layer. It serves as an encyclopedia for future modular expansions.

UI2. UI Architecture & Presentation Layer

The entire visual layer of Q-base is strictly declarative, built using Jetpack Compose and adhering rigidly to Material Design 3 (M3) guidelines. No antiquated XML layouts exist within this project. The state is strictly separated utilizing unidirectional data flow (UDF) patterns. The app responds at an uninterrupted 120 frames per second thanks to heavy coroutine orchestration preventing Main Thread locking.

2.1 Navigation & Routing

The application employs AndroidX Navigation Compose 3. The MainActivity.kt acts strictly as a host container, immediately deferring to a NavHost that controls the backstack. Routes are defined securely using Kotlin sealed classes to prevent String-based parameter injection vulnerabilities.

Routes handle complex object passing via JSON serialization (handled by kotlinx.serialization) embedded into URI arguments, specifically for deep-linking into specific StudySession configurations or parsing shared Collection URIs.

2.2 Core Screens Deep Dive

Dashboard.kt

ui/dashboard/Dashboard.kt

The Dashboard serves as the command center. It leverages kotlinx.coroutines.flow.combine inside DashboardViewModel to simultaneously observe totalCollections from CollectionDao, compute studyMinutes across all Session_Attempts, and aggregate the metric points into visual Heatmap components mimicking GitHub's contribution graphs. The heatmap uses custom Canvas drawing APIs for optimized sub-pixel anti-aliasing rendering, mapping absolute 0 indices to transparent hues, and heavily saturated MaterialTheme.colorScheme.primary blocks to represent peak study volume days.

These dual screens represent the hierarchical taxonomy of the user's data. CollectionsScreen lists top-level subjects (e.g., 'Cardiology'). The CollectionViewModel utilizes copy() mutations on data classes to reflect momentary UI states (like expanding floating action bars or engaging multi-select tracking).

StudySessionScreen.kt

ui/study/StudySessionScreen.kt

A masterclass in state machine implementation. The screen defines distinct visual compositions based on StudySessionState: Loading, ActiveQuestion, EvaluatingAnswer, and SummarySummary. When evaluating, it renders the QuestionCard composable which dynamically scales text size using custom modifiers based on the string length to prevent overflow cutting. The `EvaluatingAnswer` state immediately cascades local SQLite updates storing the elapsed milliseconds and Boolean success vector simultaneously.

ui/chat/ChatListScreen.kt | ChatDetailScreen.kt

The social hubs. DATA3. Data Persistence & State Management

Network latency is unacceptable in high-stakes study environments. Q-base executes an authoritative Offline-First architecture. UI screens read exclusively from SQLite bindings. Network modules synchronize the SQLite store retroactively. Every piece of user metadata exists fully intact without an internet connection.

The latest schema bump executed via MIGRATION_12_13 in AppDatabase.Companion executed ALTER TABLE `users` ADD COLUMN `publicKey` TEXT. This allows the local cache to resolve cryptographic keymaps rapidly without pinging Firestore for every outgoing message envelope.

3.2 Data Access Objects (DAOs)

DAOs are heavily optimized with @Transaction annotations when writing deeply nested JSON imports encompassing Questions + Options + Answers + CrossRefs to prevent partial database corruption. @Query statements employ explicit indexing (e.g., CREATE INDEX IF NOT EXISTS... in migrations) ensuring Lookups on messageId across millions of rows execute in sub-millisecond timelines. `Room` compiles these down into highly tuned C-layer SQLite bindings during the Gradle KSP phase.

3.3 ViewModel Architecture

ViewModels like ChatViewModel bind tightly to HiltViewModel for dependency scoping preventing memory leaks upon Screen rotation. They expose private val _state = MutableStateFlow() exposing val state = _state.asStateFlow() to guarantee immutability boundaries to the Compose tree. Asynchronous jobs utilize viewModelScope.launch { Dispatchers.IO } specifically protecting the Main Thread from heavy JSON Serialization or Appwrite Stream reading.

Furthermore, `BaseAiViewModel.kt` sets a strict architectural standard. It provisions pre-built coroutine exceptions handlers preventing a failed API call from crashing the application loop. Any `CancellationExceptions` are elegantly intercepted and reported as a Toast error rather than bubbling up to the `UncaughtExceptionHandler`.

3.4 DataStore Preferences

BrainDataStoreManager.kt operates asynchronously on a Protobuf-backed file via Android's DataStore primitive. It persists intricate dynamic states like TASK_CONFIGS_JSON using kotlinx.serialization.json.Json.encodeToString<Map<BrainTask, TaskConfig>>(...). This is pivotal: room schemas require compile-time rigidity, whereas DataStore easily absorbs dynamic dictionary extensions as new BrainTask parameters are birthed. It manages the absolute `isMasterAiFreeze` setting capable of blocking outbound network sockets across all AI providers simultaneously.

AI4. The AI "Brain Engine" Capability

Q-base wraps standard Google Generative AI SDKs in a dramatically sophisticated abstraction layer named the Brain Engine. This ensures the UI is entirely decoupled from SDK specifics, permitting future insertion of other LLM providers (Anthropic, DeepSeek, local ONNX models) via the BrainProvider enum architecture.

4.1 AiBrainManager Internals

brain/AiBrainManager.kt

This Singleton class orchestrates the physical HTTP dialogue. Key workflow:

  1. Receives generative request labeled with a specific BrainTask enumeration (e.g., `COLLECTION_GEN`).
  2. Queries `BrainDataStoreManager` utilizing `firstOrNull()` retrieving exactly the user's specific override preferences (Temperature boundaries, Fallback overrides) for that exact Task.
  3. Checks the global isMasterAiFreeze kill-switch; halts instantly with `AiState.Error("Brain is frozen")` and `CancellationException` if active.
  4. Initializes the GenerativeModel object explicitly wiring the System Instruction. This acts as the unbendable command framework. For collection generation, it forces pure JSON schema output without conversational preamble.
  5. Awaits the asynchronous SDK `generateContent()` resolution.
  6. Calculates `tokensEstimated` based on standard subword heuristics over the input + output strings using local Tiktoken approximations.
  7. Writes the success code, token tally, timestamp, and model hash (e.g., `gemini-1.5-flash-8b`) to Room via BrainUsageDao.insertUsage().

4.3 Telemetry & Fallback Mechanics

Failure in Generative APIs is expected (e.g. 503 Unavailable, 429 Quota Exceeded). The `AiBrainManager` implements dynamic catching mechanics: if `gemini-1.5-pro` throws a threshold block, the manager falls gracefully back to the lighter, faster `gemini-1.5-flash` model, overriding localized configurations ensuring the user experience never hard-crashes. The failure cascade is logged into BrainUsageHistory allowing developers to chart API reliability in aggregate via built-in dashboard analytics.

CLOUD5. Cloud Sync & Networking Integration

A sophisticated hybrid topography blending high-velocity Firebase NoSQL document synchronization with raw binary block capability provided by an independent Appwrite bucket instance. This guarantees Firebase throughput isn't exhausted by massive payloads.

5.1 Firebase Authentication & Firestore

AuthRepository.kt intercepts FirebaseAuth.AuthStateListener channeling login cascades into Kotlin Flows. Successful logins trigger ProfileRepository.syncUserProfile(userId) which calculates Friend Codes (`QBS-ABCD-1234`) resolving identity collisions mathematically using securely hashed UIDs appended to pseudo-random alphabetic arrays.

Firestore houses the chats collection. SyncRepository.observeAndSyncMessages binds a addSnapshotListener to chats/{id}/messages. This keeps the Local Room `MessageEntity` pool precisely equivalent to Cloud State natively in the IO dispatcher pool without starving the UI thread. Snapshot listener cancellation handles are diligently managed overriding memory leak probabilities entirely.

5.2 Appwrite Storage for ZIP Collections

To bypass massive string serialization lag inside Firestore documents, Universal Collections are processed by MockExporter.kt into discrete physical Java .zip Archive Files containing data.json manifests alongside any multimedia attachments. These are shipped to storage.createFile() using the Appwrite endpoint bound by APPWRITE_PROJECT_ID. Appwrite replies with a permanent Download URL referencing the specific chunk location. No database state rests on Appwrite, acting cleanly as a pure stateless block store.

5.3 SyncRepository Orchestration

data/repository/SyncRepository.kt

The ultimate aggregator logic controlling inbound and outbound state. When sending a message, it doesn't just hit an API. The exact execution chain entails:

  1. Filters target `participants` from `ChatDao`.
  2. Resolves target public keys actively queried against the `UserDao` or `Firestore`.
  3. Calls `CryptoManager` passing the message plaintext, producing an `encryptedPayloads` mutable map representing $N$ individually encrypted texts (where $N = $ number of users in the chat).
  4. Replaces the base `message.payload` with the stub string `"ENCRYPTED_MESSAGE"`, blinding the visual layer.
  5. Uploads the multi-keyed map to the Firestore chats/messages document payload.
  6. Firestore triggers SnapshotListener callbacks. Target devices read the multi-mapped payloads, matching their Local userId to the dictionary key mapping to their specifically wrapped ECIES ciphertext.

Firestore Rules Hierarchy (Authorization & Masking)

match /chats/{chatId}/messages/{msgId} {
  allow read: if request.auth.uid in get(/databases/$(database)/documents/chats/$(chatId)).data.participantIds;
  allow write: if request.auth.uid in get(/databases/$(database)/documents/chats/$(chatId)).data.participantIds;
}

These rigid boundaries guarantee absolute backend security isolating message reads STRICTLY to explicitly enumerated participant list strings. Even if an adversary extracts a valid JWT Firebase Bearer Token, they cannot execute Document Lookups unless their UID operates explicitly inside the `participantIds` array.

SECURITY6. Cryptography & End-to-End Encryption (E2EE)

Zero-trust architecture implemented aggressively across the communication transport layers. Q-base guarantees that even root server admins accessing Firebase/Appwrite consoles physically cannot read internal medical discussions, ensuring full HIPAA-ready compatibility regarding shared patient contexts.

6.1 Google Tink Integration

data/util/CryptoManager.kt

Implemented natively over com.google.crypto.tink:tink-android:1.8.0. Bypassing the raw, error-prone complexities of ancient javax.crypto, Tink provides safe, robust primitives actively resisting nonce collisions, repeated Initialization Vectors (IV reuse attacks), and notoriously lethal improper padding oracles. Using Tink forces the application to adopt perfectly randomized algorithmic setups.

6.2 Asymmetric Chat Encryption (ECIES)

The Elliptic Curve Integrated Encryption Scheme (ECIES). The absolute standard for multi-party chat architecture.

  • Initialization Phase: Upon a user registering (via ProfileRepository.createOrUpdateProfile), the device requests Android's Hardware Keystore (`android-keystore://qbase_master_key`) to generate a new ECIES-P256-HKDF-HMAC-SHA256-AES128-GCM template string. This constructs a physical key pair deeply tied to the physical silicon (TrustZone environment) of the device.
  • Key Transmission: The Private Key absolutely NEVER leaves the hardware layer. Any attempt to serialize it fails. The Public Key, however, is deliberately serialized to a Base64 string and pushed transparently to Firestore under the globally accessible users/{uid}.publicKey field.
  • Execution Pipeline: When User A texts User B: User A pulls B's stored `publicKey`. The device leverages a `CleartextKeysetHandle` transforming the string back into a `HybridEncrypt` interface. The `HybridEncrypt` primitive natively generates an ephemeral Elliptic Curve key, mixes it mathematically via a Hash-based Key Derivation Function (HKDF), encrypts the exact payload using standard AES-GCM, and attaches the ephemeral public key directly into the ciphertext header.
  • Decryption Flow: User B receives the Ciphertext chunk. B invokes `HybridDecrypt`. The Tink library detects the ephemeral key in the header, passes B's silicon-bound Private Key alongside it returning the underlying bytes.

6.3 Symmetric Universal File Wrapping (AES-GCM)

Tink's Asymmetric Hybrid primitives possess severe payload size limitations structurally unsuited for handling megabytes of compressed JSON assets representing thousands of Medical flashcards inside the Appwrite system.

  • File Sealing Protocol: `CryptoManager.encryptFileContent(ByteArray)` bypasses Hybrid keys completely. Instead, it forcefully requests a standalone, perfectly random `AES256_GCM` Keysethandle. Using the resulting Aead class, it encrypts the full zip byte array instantly. This handles massive byte blobs flawlessly avoiding heap overflow exceptions by utilizing internal Stream chunking where appropriate.
  • Key Delimitation Transmission: The raw, absolutely random ciphertext is pushed directly into Appwrite buckets. The highly secret, one-time transient Symmetric Key is serialized independently into Base64. It is dynamically appended to the Appwrite Download URL string utilizing a precise delimiter architecture: $downloadUrl|E2EE_KEY|$symmetricKey.
  • Ultimate Network Protection: Because the delimiter string ITSELF is passed recursively as the "TEXT" payload into the Asymmetric ECIES pipeline described in Section 6.2, the Symmetric Vault Key is flawlessly veiled in transit. The Appwrite bucket acts entirely deaf to the contents it hosts. Firebase holds an encrypted pointer holding an encrypted AES key pointing to an encrypted Appwrite bucket. It represents a flawless cryptographic barrier.
  • Decryption Target Action: `MockDownloader.downloadAndImportMock()` parses the `|E2EE_KEY|` delimiter. It requests the raw bytes via OkHttp. It passes the raw ZIP byte stream alongside the split Key directly into `CryptoManager.decryptFileContent()`, instantly returning the pristine ZIP archive into local application memory. The ciphertext never touches unencrypted local flash storage cache directories! Total obfuscation from physical forensics.
